# Spec Output

Use this reference when writing the final spec.

## Location

Write the spec to:
- `specs/<feature-name>.md`

Choose a concise, stable, hyphenated file name derived from the feature.

## Content Guidance

The spec should usually include:
- a short summary;
- goals;
- non-goals;
- current behavior or baseline if relevant;
- proposed behavior;
- acceptance criteria;
- validation plan;
- key risks or open implementation choices when they materially affect execution.

Use only the sections that help the implementation. Do not inflate the spec with boilerplate.

## Quality Bar

The spec should be:
- concrete enough to implement;
- scoped enough to avoid unrelated cleanup;
- explicit about validation expectations;
- clear about behavior-preserving vs behavior-changing work.
